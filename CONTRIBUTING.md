# CONTRIBUTING.md

Руководство для участников репозитория API-контрактов `BALLUUNN/startgrowingup-contracts`.

Цель этого файла — помочь быстро и безопасно вносить изменения в protobuf-контракты, не ломая совместимость и не нарушая правила генерации кода.

---

## 1. Что важно знать перед началом

- Этот репозиторий является источником истины для API-контрактов.
- Изменения в `.proto` считаются первичными, а сгенерированный код должен обновляться вслед за ними.
- Директорию `gen/` нельзя редактировать вручную.
- Любое изменение должно сохранять обратную совместимость, если не запланирован новый major-релиз.
- Перед началом работы обязательно ознакомьтесь с [STYLE.md](STYLE.md).

---

## 2. Рекомендуемый порядок работы

1. Найдите нужный домен и версию API в `proto/`.
2. Внесите изменение в `.proto`.
3. Проверьте совместимость и форматирование.
4. Сгенерируйте код.
5. Убедитесь, что в `gen/` нет лишних изменений.
6. Подготовьте Pull Request.

Если изменение несовместимо, создавайте новую версию API, а не модифицируйте существующую.

---

## 3. Перед изменениями

Проверьте следующее:

- новый контракт действительно нужен;
- изменение нельзя сделать через существующее поле или RPC;
- тип можно переиспользовать из `common/`, если он общий для нескольких доменов;
- имя файла, пакета, сообщения и RPC соответствует стилю репозитория;
- для Go и других целевых языков указаны необходимые options, например `go_package`.

Если вы сомневаетесь, лучше сначала расширить существующий контракт минимальным способом, чем создавать новый без необходимости.

---

## 4. Правила для `.proto`

- Используйте `snake_case` для полей и файлов.
- Используйте `PascalCase` для сообщений, сервисов и enum.
- RPC-запросы должны оканчиваться на `Request`, ответы - на `Response`.
- Все публичные поля, сообщения и RPC должны быть задокументированы.
- Комментарии должны быть на английском языке.
- Удаляемые поля сначала помечайте как `reserved`, а deprecated-поля помечайте соответствующим protobuf-атрибутом.

Для общих рекомендаций по стилю см. [STYLE.md](STYLE.md).

---

## 5. Размер и структура контрактов

- Не раздувайте сообщения без необходимости.
- Для больших наборов данных используйте пагинацию.
- Для бинарных значений используйте `bytes` только там, где это действительно уместно.
- Используйте `oneof` только для взаимоисключающих вариантов одного смысла.
- Используйте `map` только для компактных ключ-значение структур, а не для больших вложенных моделей.

Если ответ становится заметно большим, сначала рассмотрите отдельный RPC, пагинацию или разбиение данных на более компактные сущности.

---

## 6. Buf и генерация

В этом репозитории используются Buf-конфигурации:

- `buf.yaml`;
- `buf.gen.yaml`.

При работе с контрактами используйте:

```bash
buf format -w
buf lint
buf breaking --against 'https://github.com/BALLUUNN/startgrowingup-contracts.git#branch=main'
buf generate
```

Если в репозитории появится несколько модулей, `buf.work.yaml` может использоваться дополнительно, но только после согласования структуры проекта.

Никогда не редактируйте файлы в `gen/` вручную.

---

## 7. Как проверять изменения локально

Минимальный локальный набор проверок:

- `buf format -w`;
- `buf lint`;
- `buf breaking` относительно основной ветки;
- генерация кода;
- проверка отсутствия незапланированных изменений в `gen/`.

Если проект использует Makefile или task-сценарии, можно запускать соответствующие команды-обертки, но итоговый результат должен быть эквивалентен проверкам Buf.

---

## 8. Pull Request

Перед созданием PR убедитесь, что:

- изменение относится к нужной версии API;
- контракт не ломает обратную совместимость;
- все новые сообщения, поля и RPC задокументированы;
- удаленные поля помечены как `reserved`;
- сгенерированный код обновлен и не содержит ручных правок;
- в PR нет лишних косметических изменений.

Хороший PR для этого репозитория обычно содержит короткое описание контракта, список затронутых файлов и краткое объяснение совместимости.

---

## 9. Коммиты

- Делайте коммиты небольшими и логически завершенными.
- Один коммит должен описывать одну понятную группу изменений.
- Не смешивайте рефакторинг, генерацию и несовместимые изменения без необходимости.

Если вы обновляете `.proto`, убедитесь, что вместе с ним обновлен и сгенерированный код, если это требуется процессом репозитория.

---

## 10. Когда нужна новая версия API

Новая major-версия нужна, если вы:

- удаляете поле без сохранения совместимого поведения;
- меняете тип или номер поля;
- переиспользуете удаленный номер поля;
- меняете protobuf package;
- меняете значение enum так, что это ломает клиентов;
- ломаете контракт существующего RPC.

В таких случаях создайте новую версию рядом с существующей и сохраняйте старую, пока потребители не мигрируют.

---

## 11. Если вы не уверены

Если изменение может повлиять на совместимость, лучше остановиться и проверить его до PR. В этом репозитории дешевле один раз аккуратно спроектировать контракт, чем потом мигрировать множество клиентов.

---

## 12. Краткий чеклист

Перед отправкой изменений проверьте, что:

- `buf format -w` выполнен;
- `buf lint` проходит;
- `buf breaking` проходит;
- `gen/` обновлен только генерацией;
- публичные поля и RPC документированы;
- нет несовместимых изменений внутри текущей версии.
