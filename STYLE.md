# STYLE.md

Руководство по стилю для репозитория API-контрактов `BALLUUNN/startgrowingup-contracts`.

Документ задает единые правила для `.proto`-файлов, общих сообщений, версионирования и генерации кода. Цель — сохранить репозиторий предсказуемым, совместимым и удобным для автоматической проверки.

---

## 1. Базовые принципы

- Каждый `.proto` должен быть читабельным без знания внутренней реализации сервиса.
- Изменения должны быть минимальными и совместимыми по умолчанию.
- Повторяющиеся структуры выносятся в `common/` только если они действительно переиспользуются между доменами.
- Нельзя оптимизировать стиль ценой нарушения обратной совместимости.
- Любой новый контракт должен быть понятен по имени файла, пакета и сообщения.

## 1.1. Инструменты Buf

- `buf.yaml` и `buf.gen.yaml` должны соответствовать правилам этого документа.
- Форматирование выполняется только через `buf format -w`.
- Проверка качества выполняется через `buf lint`.
- Для `buf lint` следует использовать профиль `STANDARD` или эквивалентный набор правил, который покрывает:
	- корректное именование;
	- отсутствие неиспользуемых импортов;
	- отсутствие конфликтов имен;
	- совместимость и консистентность схемы;
	- требования к package и файлам.
- Изменения в конфигурации Buf должны быть согласованы с контрактами, а не существовать отдельно от них.

Минимальный ориентир для конфигурации:

```yaml
# buf.yaml
version: v2
modules:
	- path: proto
lint:
	use:
		- STANDARD
breaking:
	use:
		- FILE
```

```yaml
# buf.gen.yaml
version: v2
clean: true
plugins:
	- remote: buf.build/protocolbuffers/go
			out: gen/go
			opt: paths=source_relative
```

## 2. Структура репозитория

- Один бизнес-домен — одна директория.
- Версия API фиксируется в пути, например `auth/v1`.
- Общие типы и сообщения размещаются в `proto/common/`.
- Каждая версия API должна быть изолирована от других версий.
- Генерируемый код создается локально в `gen/` только для проверки и не редактируется вручную.

Рекомендуемая схема:

```text
proto/
	auth/v1/
	user/v1/
	search/v1/
	payment/v1/
	common/
gen/
```

## 3. Именование

### Файлы

- Имена файлов должны быть короткими, однозначными и предметными.
- Используйте `snake_case.proto`.
- Название файла должно отражать домен или тип сущности, например `session.proto`, `pagination.proto`, `errors.proto`.

### Пакеты

- `package` должен строго соответствовать пути относительно `proto/`.
- Правило маппинга: `proto/auth/v1/` -> `package auth.v1;`.
- Нельзя менять `package` в рамках существующей версии.
- Для новых версий создается новый пакет, а не модифицируется старый.

### Опции пакетов

- Для Go обязательно указывать `option go_package = "..."`.
- Путь в `go_package` должен однозначно отражать модуль и пакет импорта.
- Для других целевых языков должны быть заданы соответствующие package options, если генерация их использует.
- Нельзя полагаться на неявные значения опций, если проект генерирует код для нескольких языков.

### Сообщения и сервисы

- Имена сообщений пишутся в `PascalCase`.
- Имена RPC и сервисов должны быть действиями предметной области, а не техническими терминами.
- Не используйте слишком общие имена вроде `Data`, `Item`, `Request`, `Response` без контекста.
- Запросы RPC должны оканчиваться на `Request`, а ответы - на `Response`.


Примеры:

```proto
message UserProfile {}

service AuthService {}
```

## 4. Правила для полей

- Поля в сообщениях пишутся в `snake_case`.
- Номера полей должны быть стабильными и никогда не переиспользоваться.
- Не меняйте тип поля после публикации версии.
- Не удаляйте поля без `reserved`.
- Не добавляйте поля, которые дублируют уже существующий смысл.
- Для необязательных данных используйте `optional`, если это поддерживается выбранной схемой и действительно нужно различать отсутствие и пустое значение.

### `oneof` и `map`

- `oneof` используйте только для взаимоисключающих вариантов одного смысла.
- Не помещайте в `oneof` несвязанные поля.
- Не используйте `oneof` как замену нормальной модели данных.
- `map` используйте для небольших структурированных наборов ключ-значение.
- Не используйте `map`, если важны порядок, повторяемость или строгая валидация отдельных элементов.
- Не храните большие или глубоко вложенные данные в `map`, если для них лучше подходит отдельное сообщение.

### `bytes`

- Используйте `bytes` для хэшей, коротких бинарных идентификаторов и небольших бинарных значений.
- Не используйте `bytes` для больших файлов или потоков данных; для этого нужен отдельный транспортный механизм.
- Если значение читается как текст, предпочитайте `string`.

Пример корректного удаления:

```proto
message User {
	reserved 5;
	reserved "email";

	string name = 1;
}
```

## 5. Совместимость

Допустимо:

- добавлять новые сообщения;
- добавлять новые RPC;
- добавлять новые поля;
- добавлять новые значения enum;
- расширять контракт без изменения существующего поведения.

Запрещено:

- удалять поля без `reserved`;
- менять номер поля;
- менять тип поля;
- переиспользовать номер удаленного поля;
- переименовывать package внутри текущей версии;
- ломать семантику существующего RPC.
- изменять числовые значения enum.

Если изменение несовместимо, нужно выпускать новую версию API, например `v2`.

---

## 6. Enum

- Первое значение enum должно быть нейтральным и безопасным, обычно `*_UNSPECIFIED`.
- Не используйте порядок enum как источник логики.
- Не изменяйте числовые значения существующих enum.
- Не переиспользуйте числовые значения удаленных enum-элементов.
- Не удаляйте значения enum, если они уже могли попасть в данные или клиентский код.
- Для устаревших значений используйте комментарий о деприкации и не переиспользуйте их.
- Если значение больше не должно использоваться, сначала пометьте его как deprecated, а удаление выполняйте только в новой major-версии.

Пример:

```proto
enum UserStatus {
	USER_STATUS_UNSPECIFIED = 0;
	USER_STATUS_ACTIVE = 1;
	USER_STATUS_BLOCKED = 2;
}
```

## 7. Сервисы и RPC

- Один RPC должен делать одну понятную операцию.
- Имена методов должны описывать действие, а не внутреннюю реализацию.
- Запрос и ответ должны быть отдельными сообщениями, даже если сейчас в них одно поле.
- Не смешивайте бизнес-логику с транспортными деталями.
- Для потоковых сценариев используйте streaming только при явной необходимости.
- Для больших списков и коллекций используйте пагинацию вместо передачи больших ответов за один вызов.
- Не проектируйте сообщения, которые могут стабильно приближаться к пределам размера gRPC без явной причины.
- Ответы должны оставаться компактными и предсказуемыми по размеру.

Примеры хороших имен:

- `CreateSession`
- `GetUserProfile`
- `ListPayments`
- `DeleteAccount`

---

## 8. Общие сообщения

- `common/` предназначен для действительно общих и стабильных структур.
- Не выносите в `common/` то, что используется только одним доменом.
- Общие типы должны быть максимально нейтральными и не зависеть от конкретного сервиса.
- Перед добавлением общего сообщения проверьте, не приведет ли это к ложной связности доменов.

Типичные кандидаты:

- пагинация;
- стандартные ошибки;
- метаданные запросов;
- общие идентификаторы и фильтры.

## 8.2. Размер сообщений

- Избегайте больших сообщений, если задачу можно решить через пагинацию, фильтры или отдельный RPC.
- Не передавайте большие бинарные payload'ы внутри обычных RPC без явной необходимости.
- Учитывайте, что у gRPC есть практические ограничения на размер сообщений; проектируйте контракты так, чтобы не упираться в них по умолчанию.
- Если сообщение растет, сначала проверьте, можно ли разбить его на более компактные доменные сущности.

## 8.1. Обработка ошибок

- Ошибки должны быть описаны в общем контракте, например в `proto/common/errors.proto`.
- Не дублируйте одинаковые структуры ошибок в отдельных доменах.
- Для ошибок используйте единый формат, который можно безопасно расширять без поломки клиентов.
- RPC не должны возвращать доменно-специфичные ошибки в несовместимом виде.
- Если для ошибки есть стандартизованное представление, переиспользуйте его вместо создания нового.

---

## 9. Комментарии и документация

- Все публичные сообщения, все поля и все RPC должны иметь короткие, полезные комментарии.
- Комментарий должен объяснять смысл, а не повторять имя поля.
- Если поведение неочевидно, зафиксируйте ограничения прямо в `.proto`.
- Не оставляйте временные заметки, TODO без контекста или внутренние сокращения, непонятные внешним пользователям.
- Комментарии должны быть на английском языке.
- Каждый `service` и каждый `rpc` должны иметь комментарий.
- Все поля в `Request` и `Response` должны быть явно задокументированы, если их смысл не очевиден из имени и типа.

Хороший комментарий отвечает на вопрос: что это значит для клиента?

---

## 10. Форматирование

- Используйте единый стиль отступов и выравнивания, принятый в protobuf-инструментах.
- Не добавляйте лишние пустые строки внутри компактных блоков.
- Держите объявления в логическом порядке: package, imports, options, messages, services.
- Импорты сортируйте по алфавиту.
- Не дробите короткие сообщения на несколько экранов без причины.
- Избегайте визуального шума.
- Длина строк должна быть ограничена 120 символами, если это не ухудшает читаемость синтаксически значимых конструкций.

---

## 11. Импорты и зависимости

- Импортируйте только то, что действительно используется.
- Не создавайте циклические зависимости между пакетами.
- Старайтесь, чтобы зависимости шли от общего к частному, а не наоборот.
- Если тип нужен нескольким доменам, сначала проверьте, можно ли его сделать частью `common/`.
- Импорты из `google/protobuf/*.proto` допустимы и должны использоваться для стандартных типов, например `Timestamp` и `Empty`.
- Не вводите собственные аналоги там, где стандартный well-known type уже решает задачу.

---

## 12. Генерация кода

- Изменения в `.proto` всегда считаются источником истины.
- Код в `gen/` обновляется только через генерацию.
- Ручные правки в `gen/` запрещены.
- После генерации результат должен быть воспроизводимым и пригодным для smoke-проверок.
- Основной артефакт публикации - модуль в BSR, а не каталог `gen/` в Git.

---

## 12.1. Эталонный файл

Ниже пример структуры хорошего `.proto`-файла для этого репозитория:

```proto
syntax = "proto3";

package auth.v1;

option go_package = "github.com/BALLUUNN/startgrowingup-contracts/gen/go/auth/v1;authv1";

import "google/protobuf/timestamp.proto";
import "google/protobuf/empty.proto";

// AuthService manages user sessions and tokens.
service AuthService {
	rpc CreateSession(CreateSessionRequest) returns (CreateSessionResponse);
	rpc HealthCheck(google.protobuf.Empty) returns (google.protobuf.Empty);
}

// CreateSessionRequest contains credentials used to create a session.
message CreateSessionRequest {
	string email = 1;
	string password = 2;
}

// CreateSessionResponse returns the created session and its expiration time.
message CreateSessionResponse {
	Session session = 1;
}

// Session represents an authenticated user session.
message Session {
	string id = 1;
	google.protobuf.Timestamp expires_at = 2;
}
```

## 13. Версионирование

- Любое несовместимое изменение требует новой major-версии API.
- Новые версии должны жить рядом со старыми, пока потребители не мигрируют.
- Не смешивайте изменения для разных версий в одном контракте.
- Семантика `v1` должна оставаться стабильной весь жизненный цикл версии.
- Каждая major-версия должна иметь свой Git tag.
- Тег должен однозначно соответствовать состоянию контрактов, опубликованному в BSR для этой версии.

---

## 14. Релизная дисциплина

- Перед PR проверьте логику изменения с точки зрения совместимости.
- Генерация и lint должны проходить без ручного вмешательства.
- Каждый релиз должен быть воспроизводимым из репозитория.
- В релиз не должны попадать незадокументированные изменения контрактов.

---

## 14.1. Deprecated-поля

- Если поле планируется удалить в следующей major-версии, сначала пометьте его как deprecated.
- Используйте protobuf-атрибут `[deprecated = true]` там, где это поддерживается схемой.
- Deprecated-поля не должны использоваться для новых сценариев.
- Удаление deprecated-поля выполняется только вместе с новой несовместимой версией.

---

## 15. Проверка перед PR

Перед отправкой изменений убедитесь, что:

- `.proto` соответствует naming convention;
- нет нарушения обратной совместимости;
- новые сообщения и RPC задокументированы;
- удаленные поля отмечены как `reserved`;
- `buf generate` проходит с remote plugins;
- изменения локально читаются и не требуют устных пояснений.

---

## 16. Что считается хорошим стилем

Хороший стиль в этом репозитории — это не просто аккуратное форматирование. Это контракт, который:

- легко читать;
- легко проверять автоматически;
- безопасно расширять;
- можно использовать как основу для нескольких сервисов;
- не создает сюрпризов для клиентов.

Если изменение делает контракт сложнее, оно должно приносить ощутимую пользу и не ломать существующих потребителей.
